# Security Audit: Full Codebase

**Date:** 2026-03-21
**Scope:** All `.ts` files in `src/` (20 files) and `plugins/` (4 files)

## Summary

The codebase has solid fundamentals — IRC input sanitization is consistent, database operations use parameterized queries, plugin API objects are frozen, and the auto-op plugin follows fail-closed design. No critical exploitable vulnerabilities found. Three warnings require attention before deploying or adding planned features.

**Findings:** 0 critical, 3 warning, 7 info

## Findings

### [WARNING] CommandHandler does not enforce permission flags

**File:** `src/command-handler.ts:88-91`
**Category:** Permissions

Commands are registered with a `flags` field (e.g., `'+n'` for `.adduser`), but `CommandHandler.execute()` never checks them — it calls the handler directly. Currently safe because only the REPL routes to the command handler, and REPL has implicit owner privileges. But when IRC-to-command-handler routing is added (the planned bridge), every IRC user would be able to run `.adduser`, `.deluser`, `.load`, `.unload`, etc.

**Remediation:** Add flag checking to `CommandHandler.execute()` before calling the handler. Accept a permissions provider in the constructor, build a hostmask from the context, and reject if the user lacks the required flags. Alternatively, enforce flags at the bridging layer when it's built — but enforcing in the handler itself is defense in depth.

```typescript
// In execute(), before calling entry.handler:
if (entry.options.flags !== '-' && ctx.source !== 'repl') {
  // Check permissions for IRC callers
  if (!this.permissions.checkFlags(entry.options.flags, ctx)) {
    ctx.reply('Permission denied.');
    return;
  }
}
```

### [WARNING] Shallow freeze on botConfig exposes NickServ password to plugins

**File:** `src/plugin-loader.ts:429`
**Category:** Plugin isolation / Credentials

`Object.freeze({ ...this.botConfig })` creates a shallow copy. The top-level object is frozen, but nested objects (`irc`, `services`, `identity`) are shared references to the original mutable config. Any plugin can:

1. **Read the NickServ password:** `api.botConfig.services.password`
2. **Mutate nested config:** `api.botConfig.services.sasl = false` would modify the bot's actual config object, potentially affecting other plugins or the Services module.

**Remediation:** Deep-freeze or deep-clone the botConfig, and redact sensitive fields:

```typescript
botConfig: Object.freeze({
  irc: Object.freeze({ ...this.botConfig.irc }),
  owner: Object.freeze({ ...this.botConfig.owner }),
  identity: Object.freeze({ ...this.botConfig.identity }),
  services: Object.freeze({
    type: this.botConfig.services.type,
    nickserv: this.botConfig.services.nickserv,
    sasl: this.botConfig.services.sasl,
    // password intentionally omitted
  }),
  database: this.botConfig.database,
  pluginDir: this.botConfig.pluginDir,
  logging: Object.freeze({ ...this.botConfig.logging }),
}),
```

### [WARNING] Plugin `.load` command allows path traversal outside plugin directory

**File:** `src/core/commands/plugin-commands.ts:39`
**Category:** Input validation

The `.load` command constructs a path from user input: `` `${pluginDir}/${name}/index.ts` ``. The `name` is not validated before path construction. `.load ../../src/index` resolves to a path outside the plugin directory. The plugin loader's `SAFE_NAME_RE` validates the *exported* plugin name, not the directory name from the command.

Mitigated by requiring `+n` (owner) flags, and the target file must export `name` + `init` to load successfully. But a malicious-looking `.ts` file elsewhere on disk could be loaded.

**Remediation:** Validate the plugin name in the `.load` command before constructing the path:

```typescript
// In plugin-commands.ts, inside the .load handler:
const name = args.trim();
if (!name || !/^[a-zA-Z0-9][a-zA-Z0-9_-]*$/.test(name)) {
  ctx.reply('Invalid plugin name. Use alphanumeric characters, hyphens, and underscores only.');
  return;
}
```

### [INFO] REPL commands are not logged to mod_log

**File:** `src/repl.ts:73`
**Category:** Credentials / Audit trail

SECURITY.md section 3.4 states "Log all REPL commands the same way IRC commands are logged." Currently, REPL commands are not logged anywhere except individual permission changes (which the Permissions class logs). Admin actions like `.load`, `.unload`, `.say` from the REPL have no audit trail.

### [INFO] `ensureOwner` does not update an existing owner's hostmask

**File:** `src/bot.ts:247-256`
**Category:** Permissions

If the owner record already exists in the database (from a previous run), changing the hostmask in `config/bot.json` has no effect — `ensureOwner()` skips the update. This caused an operational issue during testing when the hostmask needed to change for Rizon's cloaking. Consider updating the hostmask if it differs from config.

### [INFO] `raw()` exposed on plugin API

**File:** `src/plugin-loader.ts:358-360`
**Category:** Plugin isolation

Plugins can send arbitrary IRC protocol lines via `api.raw()`. This is by design (matching Eggdrop's API), but means a buggy or malicious plugin can send any IRC command, bypass newline sanitization, or impersonate the bot.

### [INFO] Seen plugin stores messages without TTL or size limits

**File:** `plugins/seen/index.ts:15-22`
**Category:** DoS

Every channel message is stored in the database with no expiration. Over months of operation in busy channels, this could grow significantly. Consider adding a periodic cleanup or TTL.

### [INFO] No output message length splitting

**File:** `src/irc-bridge.ts:340-341` (ctx.reply)
**Category:** IRC-specific

`ctx.reply()` sends the full message via `client.say()`. IRC messages are limited to ~512 bytes including protocol overhead. Long responses are silently truncated by the server rather than split at word boundaries. This could produce garbled output for plugins that generate long replies (e.g., `.users` with many users, or `!seen` echoing a long stored message).

### [INFO] RFC 2812 nick case mapping not fully implemented

**File:** `src/core/channel-state.ts` (various)
**Category:** IRC-specific

Nick lookups use JavaScript's `.toLowerCase()`, which doesn't handle the IRC-specific case mapping where `{`, `}`, `|` are the lowercase equivalents of `[`, `]`, `\` (per RFC 2812 / ISUPPORT CASEMAPPING). In practice this rarely causes issues since most nicks use standard ASCII, but it could cause lookup failures for nicks containing these characters.

### [INFO] `findByHostmask` returns full UserRecord to plugins

**File:** `src/plugin-loader.ts:318-319`
**Category:** Plugin isolation

The permissions API exposes `findByHostmask()` which returns the complete `UserRecord` including all hostmask patterns, global flags, and per-channel flags. This is necessary for auto-op but means any plugin can enumerate user data. Not a vulnerability given the trust model (admin installs plugins), but worth noting for defense in depth.

### [INFO] Timer binds have no minimum interval guard

**File:** `src/dispatcher.ts:78-108`
**Category:** DoS

A plugin can register a timer bind with a very short interval (e.g., 1 second) that floods the channel, potentially causing the bot to be disconnected or banned. The admin can unload the plugin, but there's no built-in rate limit.

## Passed Checks

- **IRC input sanitization:** All event fields pass through `sanitize()` in irc-bridge.ts (strips `\r\n`)
- **IRC formatting stripped:** Command text stripped of bold/color/underline before parsing
- **Database parameterization:** All queries use prepared statements via better-sqlite3
- **Plugin name validation:** `SAFE_NAME_RE` rejects names with path separators or special characters
- **Plugin init error containment:** `init()` wrapped in try/catch; failed binds cleaned up
- **Plugin teardown error containment:** `teardown()` wrapped in try/catch
- **Plugin API frozen:** `Object.freeze()` on the API object and nested db/permissions/services
- **Database namespace isolation:** Enforced at the Database class level, not by convention
- **Auto-op hostmask-first design:** Hostmask checked before NickServ query (doesn't ACC-check every joiner)
- **Auto-op fail-closed:** Verification timeout → mode NOT applied
- **Auto-op skip-self:** Bot doesn't attempt to op itself
- **Insecure hostmask warning:** `nick!*@*` patterns flagged with `[security]` log for privileged users
- **Config credentials gitignored:** `config/bot.json` and `config/plugins.json` in `.gitignore`
- **Example configs clean:** No real credentials in `bot.example.json`
- **SASL password not logged:** `services.identify()` logs "Sent IDENTIFY" without the password
- **Newlines stripped in kick/topic:** `irc-commands.ts` strips `\r\n` from reason and topic text
- **MODES batching:** IRCCommands respects ISUPPORT MODES limit

## Recommendations

1. **Fix WARNING items before deploying to a public network.** The shallow botConfig freeze is the most impactful — fix it now since plugins are already loading.
2. **Add flag enforcement to CommandHandler before building IRC→command routing.** This is not exploitable today but will become critical the moment that bridge is added.
3. **Validate plugin names in the `.load` command** — simple one-line fix.
4. **Consider adding a `sanitize()` wrapper for plugin `raw()` calls** — strip `\r\n` from the line before sending to prevent accidental protocol injection by buggy plugins.
