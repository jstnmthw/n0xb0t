# Security Audit: Full Codebase

**Date:** 2026-03-22
**Scope:** All `.ts` files in `src/` (23 files) and `plugins/` (6 files)

## Summary

The codebase demonstrates strong security practices overall. The IRC bridge properly sanitizes all incoming data at the trust boundary, database access uses parameterized queries exclusively, the plugin API is frozen with namespaced DB access and redacted credentials, and the dispatcher correctly enforces flag-based permissions before calling handlers. The NickServ verification race condition is properly handled in the chanmod plugin.

Five warnings and six info items were identified — none are immediately exploitable from an external IRC user, but they represent gaps between the security guide's stated practices and the implementation.

**Findings:** 0 critical, 5 warning, 6 info

## Findings

### [WARNING] IRC protocol injection via `raw()` in IRCCommands

**File:** `src/core/irc-commands.ts:68-70, 56-58, 104, 143-145, 147-152`
**Category:** Protocol injection / Defense-in-depth

**Description:** Multiple methods in `IRCCommands` interpolate `channel`, `nick`, `mask`, and `key` parameters directly into `client.raw()` calls without sanitization:

```typescript
// kick() — channel and nick unsanitized
this.client.raw(`KICK ${channel} ${nick} :${safe}`);

// join() with key — both unsanitized
this.client.raw(`JOIN ${channel} ${key}`);

// topic() — channel unsanitized
this.client.raw(`TOPIC ${channel} :${safe}`);

// sendMode() fallback — all unsanitized
this.client.raw(`MODE ${channel} ${mode} ${param}`);

// sendModeRaw() — all unsanitized
this.client.raw(`MODE ${channel} ${modeString} ${params.join(' ')}`);
```

In normal operation these values come from the plugin API (which receives already-sanitized data from the IRC bridge) or from validated bot commands, so injection is unlikely. However, the plugin API methods (`op`, `deop`, `kick`, `ban`, `mode`, `topic`) pass arguments straight through to `IRCCommands` without sanitization (see `src/plugin-loader.ts:375-397`), while the IRC output methods (`say`, `notice`, `action`, `raw`) do sanitize. This inconsistency means a buggy plugin that constructs a channel or nick from user input without sanitizing could inject arbitrary IRC commands.

**Remediation:** Sanitize `channel`, `nick`, `mask`, `key`, and `modeString` parameters in `IRCCommands` methods before interpolating into `raw()` calls, just as `reason` and `text` are already sanitized. Alternatively, sanitize at the plugin API layer in `createPluginApi()` for `op`, `deop`, `voice`, `devoice`, `kick`, `ban`, `mode`, and `topic`.

### [WARNING] Case-sensitive channel flag lookups in permissions

**File:** `src/core/permissions.ts:294-301`
**Category:** Permissions

**Description:** Channel-specific flag lookups use exact string matching on the channel name:

```typescript
private userHasFlag(record: UserRecord, flag: string, channel: string | null): boolean {
  // ...
  if (channel) {
    const channelFlags = record.channels[channel]; // exact key lookup
    // ...
  }
}
```

IRC channel names are case-insensitive (RFC 1459 CASEMAPPING). If an admin sets `.flags user +o #MyChannel` but the dispatcher later checks with `#mychannel` (as normalized by the server), the channel flag lookup silently fails. The `setChannelFlags` method also stores the channel name as-is. This could cause users to unexpectedly lack permissions in specific channels despite having been granted them.

**Remediation:** Apply `ircLower()` to channel names when storing (`setChannelFlags`) and when looking up (`userHasFlag`). Example:

```typescript
// In setChannelFlags:
const normalizedChannel = ircLower(channel);
record.channels[normalizedChannel] = normalized;

// In userHasFlag:
const channelFlags = record.channels[ircLower(channel)];
```

### [WARNING] Kick event context mixes kicker's identity with kicked nick

**File:** `src/irc-bridge.ts:244-267`
**Category:** Input validation / Permissions

**Description:** The `onKick` handler builds a context where `ctx.nick` is the kicked user but `ctx.ident` and `ctx.hostname` belong to the *kicker*:

```typescript
private onKick(event: Record<string, unknown>): void {
  const kicker = sanitize(String(event.nick ?? ''));    // the user who kicked
  const ident = sanitize(String(event.ident ?? ''));    // kicker's ident
  const hostname = sanitize(String(event.hostname ?? '')); // kicker's hostname
  const kicked = sanitize(String(event.kicked ?? ''));  // the kicked user

  const ctx = this.buildContext({
    nick: kicked,   // kicked user's nick
    ident,          // ← kicker's ident (WRONG for the kicked user)
    hostname,       // ← kicker's hostname (WRONG for the kicked user)
    ...
  });
}
```

Any `kick` bind handler that attempts to resolve the kicked user's hostmask via `${ctx.nick}!${ctx.ident}@${ctx.hostname}` would produce `kicked_nick!kicker_ident@kicker_hostname` — an invalid, mixed hostmask. For example, a plugin that auto-invites kicked ops back based on their permissions would silently fail to resolve the correct user record.

**Remediation:** Put both identities into the context properly. The cleanest approach is to look up the kicked user's hostmask from channel state (which has the correct ident/hostname) and populate the context from that:

```typescript
// Option A: use channel state for the kicked user's full hostmask
const kickedHostmask = channelState?.getUserHostmask(channel, kicked) ?? '';
const [kickedIdent, kickedHost] = parseHostmask(kickedHostmask);
const ctx = this.buildContext({ nick: kicked, ident: kickedIdent, hostname: kickedHost, ... });
```

If channel state isn't available in the bridge, at minimum document that `ctx.ident`/`ctx.hostname` in kick events are the kicker's, not the kicked user's.

### [WARNING] Missing config file permission check on startup

**File:** `src/bot.ts:260-277`
**Category:** Credentials

**Description:** `docs/SECURITY.md` Section 6 states: "The bot should refuse to start if `config/bot.json` is world-readable (`chmod` check on startup, at least on Unix)." The current `loadConfig()` only checks whether the file exists and is readable (`accessSync(configPath, fsConstants.R_OK)`), not whether it has overly permissive file modes. On a shared system, a world-readable `bot.json` exposes the NickServ/SASL password.

**Remediation:** Add a `statSync` check after opening the config file:

```typescript
import { statSync } from 'node:fs';

const stat = statSync(cfgPath);
const worldReadable = stat.mode & 0o004;
if (worldReadable) {
  console.error(`[bot] SECURITY: ${cfgPath} is world-readable (mode ${(stat.mode & 0o777).toString(8)})`);
  console.error('[bot] Run: chmod 600 ' + cfgPath);
  process.exit(1);
}
```

### [WARNING] Shallow freeze of botConfig exposes mutable arrays to plugins

**File:** `src/plugin-loader.ts:444-457`
**Category:** Plugin isolation

**Description:** The `botConfig` object exposed to plugins uses `Object.freeze({ ...this.botConfig.irc })`, which creates a shallow copy. The `channels` array within the `irc` config is the same reference as the original — `Object.freeze` prevents reassignment of the `channels` property but does not freeze the array itself. A plugin can mutate it:

```typescript
// In a malicious/buggy plugin:
(api.botConfig.irc as any).channels.push('#evil');
```

This would modify the underlying config object visible to all other plugins.

**Remediation:** Deep-freeze the channels array, or copy it:

```typescript
irc: Object.freeze({
  ...this.botConfig.irc,
  channels: Object.freeze([...this.botConfig.irc.channels]),
}),
```

### [INFO] No rate limiting on CTCP responses

**File:** `src/irc-bridge.ts:127-142`
**Category:** DoS

**Description:** The built-in CTCP handlers for VERSION, PING, and TIME respond to every incoming request with no rate limiting. An attacker can spam CTCP requests from multiple nicks, causing the bot to queue many responses and potentially flood the server.

**Remediation:** Add per-nick or global rate limiting to CTCP response handlers (e.g., max 3 responses per 10 seconds per nick).

### [INFO] `.say` command does not validate target format

**File:** `src/core/commands/irc-commands-admin.ts:44,51`
**Category:** Input validation

**Description:** The `.say` command sanitizes the message body but does not validate the target parameter format. While the command is protected by `+o` flags and the input is already sanitized (newlines stripped by the IRC bridge for IRC sources; readline for REPL), the target is not checked to be a valid channel or nick pattern. The SECURITY.md guide (Section 2.2) recommends validating targets.

**Remediation:** Add a lightweight target validation:

```typescript
if (!/^[#&]?\w/.test(target)) {
  ctx.reply('Invalid target.');
  return;
}
```

### [INFO] `topic` plugin: `String.replace()` with user input allows JS pattern substitution

**File:** `plugins/topic/index.ts:42, 60`
**Category:** Input validation

**Description:** The topic formatting uses:

```typescript
const formatted = template.replace('$text', text);
```

JavaScript's `String.prototype.replace()` treats certain sequences in the *replacement string* as special patterns even when the search pattern is a plain string: `$&` inserts the matched text (`$text`), `$'` inserts the text after the match, `` $` `` inserts the text before the match. If a user passes text containing these sequences, the formatted output is unexpected. For example:

```
!topic rune $& injected
→ template: "... $text ..."
→ formatted: "... $text injected ..."   (topic contains literal "$text" again)
```

This cannot inject IRC commands (the bot's `sanitize()` catches `\r\n`), but it produces garbled or doubled topic content.

**Remediation:** Use a function replacement to avoid pattern interpretation:

```typescript
const formatted = template.replace('$text', () => text);
```

### [INFO] Greeter plugin does not strip IRC formatting codes from nick

**File:** `plugins/greeter/index.ts:23-26`
**Category:** Output safety

**Description:** The greeter interpolates `ctx.nick` directly into the greeting string:

```typescript
const greeting = message
  .replace(/\{channel\}/g, ctx.channel ?? '')
  .replace(/\{nick\}/g, ctx.nick);
ctx.reply(greeting);
```

The bridge's `sanitize()` only strips `\r\n` — IRC formatting control characters (`\x02` bold, `\x03` color, `\x1F` underline, etc.) are preserved. If a server permits nicks containing these characters, the greeting could carry formatting codes that mislead users reading the channel — for example, colored text that mimics a system message or hides part of the greeting.

**Remediation:** Strip IRC formatting from the nick before interpolation. The `stripFormatting()` helper already exists in `src/irc-bridge.ts` (private) — export it or re-implement a version in a shared utility:

```typescript
const safeNick = ctx.nick.replace(/[\x02\x03\x0F\x16\x1D\x1E\x1F]|\x03\d{1,2}(,\d{1,2})?/g, '');
```

### [INFO] Message queue is unbounded

**File:** `src/core/message-queue.ts:26`
**Category:** DoS

**Description:** `MessageQueue.queue` is an unbounded array. Under sustained conditions where handlers enqueue messages faster than the token bucket drains them (e.g., a `pubm` handler that replies to every channel message during a bot flood), the queue grows without limit, consuming memory until the process is killed.

**Remediation:** Add a maximum queue depth with a discard (or error) policy:

```typescript
const MAX_QUEUE_DEPTH = 500;

enqueue(fn: () => void): void {
  // ...
  if (this.queue.length >= MAX_QUEUE_DEPTH) {
    this.logger?.warn(`Message queue full (${MAX_QUEUE_DEPTH}), dropping message`);
    return;
  }
  this.queue.push(fn);
}
```

## Passed checks

- **IRC bridge sanitization** — All incoming IRC data passes through `sanitize()` in `irc-bridge.ts`, stripping `\r\n` before any dispatcher handling
- **IRC formatting stripping** — Command parsing strips bold/color/underline control characters before extracting command names
- **Channel name validation** — `isValidChannel()` gate on all channel-targeted dispatches
- **Database parameterization** — All KV and mod_log queries use `better-sqlite3` prepared statements with `?` placeholders
- **`getModLog` dynamic SQL** — WHERE clause structure is assembled from hardcoded strings; values use `?` parameters; not vulnerable to SQL injection
- **Plugin namespace isolation** — Database operations are scoped by `pluginId`; plugins cannot access other namespaces
- **Plugin API frozen** — `Object.freeze(api)` prevents plugins from modifying the API object
- **Plugin DB/permissions/services frozen** — Sub-objects are individually frozen
- **Credential redaction** — `services.password` is omitted from plugin-facing `botConfig`
- **Password never logged** — NickServ IDENTIFY sends password but logs only "Sent IDENTIFY to NickServ"
- **NickServ debug logging safe** — Logs NickServ response text (which doesn't contain passwords) at debug level only
- **Config file gitignored** — `config/bot.json` and `config/plugins.json` are in `.gitignore`
- **Example configs credential-free** — `bot.example.json` has `"password": ""`, no real credentials
- **Insecure hostmask warning** — `permissions.ts` warns on `nick!*@*` patterns for privileged users
- **NickServ ACC race handled** — chanmod awaits `verifyUser()` before opping when `require_acc_for` is configured
- **Verification timeout** — NickServ verification has a 5-second timeout, preventing indefinite hangs
- **Flag checking fail-closed** — Unknown user returns `false` from `checkFlags()`; no permissions provider defaults to allow-all (but see dispatcher note below)
- **Timer minimum interval** — Dispatcher enforces 10-second minimum on `time` binds
- **Message splitting and line cap** — `splitMessage` caps output at 4 lines with 400-byte line limit
- **Plugin path traversal prevention** — `.load` command validates name against `^[a-zA-Z0-9][a-zA-Z0-9_-]*$`
- **Plugin error containment** — Dispatcher wraps handlers in try/catch; plugin loader catches init()/teardown() errors
- **Plugin resource cleanup** — `unbindAll()` clears all binds and timers; teardown errors don't block unload
- **Reply sanitization** — `buildContext` in IRC bridge sanitizes reply output via `sanitize(msg)` before sending
- **Owner implied flags** — Owner flag `n` correctly implies all other flags via `userHasFlag()`
- **chanmod input validation** — `isValidNick()` rejects nicks with `\r\n\s`; ban masks checked for newlines
- **chanmod bot self-protection** — Prevents self-deop, self-kick, self-ban
- **chanmod ops check** — All moderation commands verify bot has ops before acting
- **Mode enforcement rate limiting** — Cooldown prevents mode wars (max 3 re-enforcements per 10 seconds)
- **IRC case-insensitive matching** — `ircLower()` used throughout dispatcher mask matching and channel state lookups
- **seen plugin text truncation** — Message text capped at 200 characters before storage

## Recommendations

1. **Fix the raw() sanitization gap and kick context bug first** — These are the two highest-priority warnings; the raw() gap is a latent injection risk for future plugins, and the kick context bug is a correctness trap that will silently corrupt any handler that checks kicked-user permissions.
2. **Fix channel flag case sensitivity** — Easy fix, prevents silent permission mismatches on networks that normalize channel names.
3. **Deep-freeze botConfig channels array** — One-line fix, closes a plugin isolation gap.
4. **Add bot.json world-readability check** — Implements the stated SECURITY.md requirement; important on shared/cloud hosts.
5. **Consider CTCP rate limiting** — Easy to add and prevents a class of flood disconnects.
6. **Fix `topic` plugin `$text` replacement** — One-line fix (`() => text` callback), removes unexpected behavior for users.
7. **Add message queue max depth** — Prevents memory exhaustion under flood conditions.
