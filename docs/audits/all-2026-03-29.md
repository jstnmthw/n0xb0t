# Security Audit: all

**Date:** 2026-03-29
**Scope:** All `.ts` files in `src/` and `plugins/`

---

## Summary

HexBot is in good security shape overall. The core trust boundary (`irc-bridge.ts`)
consistently sanitizes all incoming IRC data. Parameterized database queries are used
throughout. The plugin API is frozen and namespaced. The dispatcher enforces flag checks
and NickServ ACC verification automatically. No `eval()` or `new Function()` calls were
found anywhere in the codebase.

Three findings are notable — none are critical, two involve IRC formatting codes passing
through to output without stripping, and one is a case-mapping inconsistency in the
auto-op plugin.

**Findings:** 0 critical, 3 warning, 4 info

---

## Findings

### [WARNING] `seen` plugin reflects user nick, channel, and text without `stripFormatting()`

**File:** `plugins/seen/index.ts:71`
**Category:** Output safety (SECURITY.md §5.2)

```typescript
ctx.reply(`${record.nick} was last seen ${ago} in ${record.channel} saying: ${record.text}`);
```

`record.nick`, `record.channel`, and `record.text` are all stored from IRC events, which
means they may contain IRC formatting codes (bold `\x02`, color `\x03`, etc.). When
replayed into the channel, a user who set their nick to a string containing IRC color codes
could make the `!seen` reply appear visually misleading to other users — e.g., embedding
color resets to hide part of the message, or making it appear the bot said something else.

**Remediation:** Apply `api.stripFormatting()` to all user-controlled fields in the reply:

```typescript
ctx.reply(
  `${api.stripFormatting(record.nick)} was last seen ${ago} ` +
    `in ${api.stripFormatting(record.channel)} saying: ${api.stripFormatting(record.text)}`,
);
```

### [WARNING] DCC announce mirrors raw IRC private messages without sanitizing

**File:** `src/core/dcc.ts:389–404`
**Category:** Output safety / protocol injection into DCC terminal

The `attach()` method registers listeners that mirror incoming IRC PMs and NOTICEs to all
active DCC sessions:

```typescript
const onNotice = (...args: unknown[]) => {
  const e = toEventObject(args[0]);
  const nick = String(e.nick ?? '');
  const message = String(e.message ?? '');
  if (/^[#&]/.test(target)) return;
  this.announce(`-${nick}- ${message}`); // no sanitize()
};
```

Neither `nick` nor `message` are passed through `sanitize()` before being written to DCC
sessions. DCC CHAT uses a raw TCP readline protocol, and the `DCCSession.writeLine()` method
appends `\r\n` to the output. If an attacker sends the bot a crafted PRIVMSG containing
embedded `\r\n`, the output written to DCC sessions will be split across multiple lines,
allowing injection of content that looks like system messages (`*** admin has joined the
console`) or other DCC user messages (`<handle> ...`) into authenticated admin sessions.

The same code exists in `src/repl.ts:47–56` but is lower-risk since the REPL is local
and `\r\n` sequences just create extra blank lines on the terminal.

**Remediation:** Sanitize both fields before announcing to DCC sessions:

```typescript
import { sanitize } from '../utils/sanitize';

// ...
this.announce(`-${sanitize(nick)}- ${sanitize(message)}`);
```

Apply the same fix to the `onPrivmsg` mirror handler on line 396–404.

### [WARNING] `auto-op.ts` looks up per-channel flags with raw channel name, not `ircLower()`

**File:** `plugins/chanmod/auto-op.ts:41`
**Category:** Identity / permissions (correctness failure)

```typescript
const channelFlags = user.channels[channel] ?? '';
```

`channel` is `ctx.channel!` from the IRC event — sanitized but not case-normalized. The
permissions system stores per-channel flag keys as `ircLower(channel, casemapping)` (see
`permissions.ts:138`). On networks that send `#Channel` from the server (some do), the
raw channel name won't match the stored key, so per-channel `+o`/`+v` flags are silently
ignored and the user falls back to global flags only.

This is a conservative failure (the user may not get auto-opped when they should), but it
is inconsistent with how every other location in the codebase looks up per-channel flags
(e.g., `helpers.ts:82` uses `api.ircLower(channel)`).

**Remediation:**

```typescript
const channelFlags = user.channels[api.ircLower(channel)] ?? '';
```

Note: `plugins/greeter/index.ts:82` has the same issue, using `channel.toLowerCase()`
instead of `api.ircLower(channel)` for per-channel flag lookups. The impact is the same
(conservative failure — user doesn't get a custom greet when they should).

### [INFO] `sanitize()` only strips `\r\n` — NUL bytes and other control chars pass through

**File:** `src/utils/sanitize.ts`
**Category:** Input validation (defense-in-depth)

```typescript
export function sanitize(text: string): string {
  return text.replace(/[\r\n]/g, '');
}
```

The function covers the most dangerous IRC injection vector (newline-delimited commands).
NUL bytes (`\0`) and other control characters pass through but `irc-framework` handles
low-level protocol framing, so this does not constitute an exploitable injection path.
However, NUL bytes in messages could cause display artifacts or unexpected behavior in
downstream consumers.

**Remediation (optional, defense-in-depth):**

```typescript
export function sanitize(text: string): string {
  return text.replace(/[\r\n\0]/g, '');
}
```

### [INFO] REPL console output of IRC PMs includes raw nick/message without control-char stripping

**File:** `src/repl.ts:47–56`
**Category:** Output safety (terminal display only)

The REPL mirrors incoming private messages to the operator's terminal:

```typescript
this.print(`-${nick}- ${message}`);
this.print(`<${nick}> ${message}`);
```

Neither `nick` nor `message` are sanitized. IRC messages can contain ANSI-like control
sequences (`\x1b[…`) that, when printed to a terminal, could clear the screen, move the
cursor, or inject false prompts. Since this only affects the local operator's terminal
(not network output), exploitation requires an attacker to already know the bot's IRC
address and send it PMs. Impact is limited to terminal UI confusion.

**Remediation:** Apply `sanitize()` (or a more aggressive strip of non-printable ASCII)
before printing to the terminal.

### [INFO] `isValidNick()` in chanmod does not validate IRC nick syntax

**File:** `plugins/chanmod/helpers.ts:31`
**Category:** Input validation

```typescript
export function isValidNick(nick: string): boolean {
  return nick.length > 0 && !/[\r\n\s]/.test(nick);
}
```

This accepts any non-empty, non-whitespace string as a valid nick — including `@nick`
(op-prefixed), `#chan` (which looks like a channel), or strings with IRC-special characters.
Passing such a value to `api.op(channel, nick)` will result in the IRC server rejecting the
MODE command rather than silently succeeding. The failure mode is benign (IRC server error,
not a security bypass), but tightening the regex would prevent wasted IRC round-trips.

**Remediation:**

```typescript
export function isValidNick(nick: string): boolean {
  return /^[a-zA-Z\[\]\\`_^{|}][a-zA-Z0-9\[\]\\`_^{|}\-]{0,49}$/.test(nick);
}
```

## Passed checks

- **Newline injection at trust boundary:** `irc-bridge.ts` calls `sanitize()` on all
  fields from every IRC event before building `HandlerContext`. The bridge is the only path
  into the dispatcher.
- **Database SQL injection:** All `better-sqlite3` queries use prepared statements with
  `?` parameters. The one dynamic query in `getModLog()` builds only from hardcoded
  condition strings.
- **Plugin namespace isolation:** `createPluginDbApi()` enforces that every plugin's
  DB calls are prefixed with the plugin ID; plugins cannot access other namespaces.
- **Plugin API freeze:** `Object.freeze(api)` is called in `createPluginApi()`. Plugins
  cannot mutate the API object they receive.
- **Flag checking before dispatch:** The dispatcher checks flags via `PermissionsProvider`
  before calling any handler, and enforces NickServ ACC via `VerificationProvider` for
  flagged binds when `require_acc_for` is configured.
- **NickServ ACC race (auto-op):** `auto-op.ts` awaits `api.services.verifyUser(nick)`
  before calling `api.op()`, and only does so when `requireAccFor.includes(flagToApply)`.
  The dispatcher's own `checkVerification()` additionally gates privileged `pub`/`msg`
  binds via the live account map (fast path) + ACC fallback.
- **Error containment:** The dispatcher wraps every handler call in `try/catch`. A
  throwing plugin does not crash the bot or block other handlers.
- **CTCP rate limiting:** `IRCBridge.ctcpAllowed()` enforces max 3 CTCP responses per
  nick per 10 seconds.
- **Flood protection:** The dispatcher's `floodCheck()` is applied per-message before
  dispatching `pub`/`pubm` and `msg`/`msgm`. Owners (`n` flag) bypass flood limits.
- **Mode war protection:** `mode-enforce.ts` tracks enforcement counts per channel/nick
  and suppresses re-enforcement after `MAX_ENFORCEMENTS` attempts.
- **Insecure hostmask warning:** `permissions.ts:warnInsecureHostmask()` logs a `SECURITY`
  warning when a privileged user is added with `nick!*@*` or `nick!*@*.*`.
- **No eval/new Function:** No dynamic code evaluation found anywhere in the codebase.
- **Credentials excluded from logs:** `services.ts` logs `'Sent IDENTIFY to NickServ'`
  without including the password. The `PluginBotConfig` exposed to plugins omits the
  `services.password` field.
- **Config world-readable check:** `bot.ts:386-393` refuses to start if `config/bot.json`
  has world-read permission set.
- **TLS enabled by default:** The example config has `"tls": true`. A warning is printed
  if `tls_verify: false` is set.
- **DCC race condition mitigations:** 30-second pending timeout, single-connection listener,
  flags + optional NickServ gate before port allocation — documented in SECURITY.md §3.4.
- **No direct IRC raw() from user input:** Plugins use typed methods (`say`, `notice`,
  `mode`, etc.). The one `raw()` call in the codebase is in `changeNick()` which applies
  `sanitize()` before use.

---

## Recommendations

1. **Address WARNING #1 and #2 (stripFormatting + DCC mirror sanitize)** — these are
   the two most actionable fixes. Both are one-line changes.

2. **Address WARNING #3 (auto-op and greeter ircLower)** — low risk but easy to fix;
   use `api.ircLower(channel)` consistently for all per-channel flag lookups.

3. **Consider tightening `isValidNick()` regex** (INFO #4) — the current permissiveness
   only causes benign IRC server rejections, but a stricter check communicates intent more
   clearly.

4. **Consider stripping NUL in `sanitize()`** (INFO #2) — one-character addition for
   better defense-in-depth.
