## Findings

### [WARNING] CTCP rate limiter is dead code — `ctcpAllowed()` never called

**File:** `src/irc-bridge.ts:119`, `src/plugin-loader.ts:414`
**Category:** DoS

**Description:** `ctcpAllowed()` was added to `IRCBridge` to enforce a max-3-per-10-seconds per-nick rate limit on CTCP responses. However, it is never called anywhere in the codebase — it is dead code.

```typescript
// src/irc-bridge.ts — defined but never invoked:
private ctcpAllowed(nick: string): boolean {
  // ...
}

// onCtcp() dispatches without any rate check:
private onCtcp(event: Record<string, unknown>): void {
  // ...
  this.dispatcher.dispatch('ctcp', ctx).catch(this.dispatchError('ctcp'));
}
```

Additionally, `api.ctcpResponse()` in `plugin-loader.ts:414` sends directly via `ircClient?.ctcpResponse()` without going through the message queue, unlike `say()`, `notice()`, and `action()` which are all queue-backed. This means CTCP responses (which are IRC NOTICEs) have no flood protection at all — neither the rate limiter nor the token-bucket queue applies.

An attacker who can send CTCP requests (any unprivileged IRC user) can trigger one bot NOTICE per request, potentially causing a flood disconnect on servers with low outbound flood limits.

**Remediation:**

Option A — gate in `onCtcp` before dispatching (prevents flood at the source):

```typescript
private onCtcp(event: Record<string, unknown>): void {
  const nick = sanitize(String(event.nick ?? ''));
  // ...
  if (!this.ctcpAllowed(nick)) return; // drop silently
  this.dispatcher.dispatch('ctcp', ctx).catch(this.dispatchError('ctcp'));
}
```

Option B — route `ctcpResponse` through the message queue (consistent with `say`/`notice`/`action`):

```typescript
// src/plugin-loader.ts
ctcpResponse(target: string, type: string, message: string): void {
  const safeTarget = sanitize(target);
  const safeType = sanitize(type);
  const safeMessage = sanitize(message);
  if (messageQueue) {
    messageQueue.enqueue(() => ircClient?.ctcpResponse(safeTarget, safeType, safeMessage));
  } else {
    ircClient?.ctcpResponse(safeTarget, safeType, safeMessage);
  }
},
```

Both options should be applied: option A rate-limits inbound CTCP bursts before dispatching handlers (cheaper), and option B ensures any remaining responses are token-bucket throttled on the way out (defense-in-depth). The `ctcpAllowed()` method already exists and is correct — it just needs to be called.

### [INFO] `raw()` bypasses message queue

**File:** `src/plugin-loader.ts:411`
**Category:** DoS / Defense-in-depth

**Description:** The plugin API `raw()` method sends immediately via `ircClient?.raw()` without going through the message queue:

```typescript
raw(line: string): void {
  ircClient?.raw(sanitize(line));
},
```

`say()`, `notice()`, and `action()` all pass through the message queue for flood protection. A buggy or malicious plugin that calls `raw()` in a loop (e.g., in a `pubm` catch-all handler) could flood the server without any throttling. No current plugin uses `raw()`, so this is theoretical.

**Remediation:** Document the gap in the type declaration and in CLAUDE.md:

```typescript
// src/types.ts — add to JSDoc:
/**
 * Send a raw IRC line. NOT rate-limited — bypasses the message queue.
 * Use only for IRC commands that have no typed equivalent.
 * Never call in a loop or from user-triggered handlers.
 */
raw(line: string): void;
```

Routing `raw()` through the queue is also an option but may break callers that need ordering guarantees relative to the IRC connection state (e.g., a raw CAP command during registration).

## Passed checks

All checks from the previous audit remain passing, plus the new areas introduced by the CASEMAPPING feature:

- **CASEMAPPING input validation** — `bot.ts` validates the server-sent value against the three known tokens before assigning; unknown values fall back to `'rfc1459'`
- **CASEMAPPING propagation** — All modules receive the active casemapping via `setCasemapping()` after `registered`; default `'rfc1459'` is safe for pre-connect state
- **Hostmask matching unchanged** — `findByHostmask()` keeps `'rfc1459'` hardcoded; hostmask patterns don't contain characters affected by casemapping differences
- **Plugin API ircLower() live closure** — `api.ircLower()` reads `getCasemapping()` at call time, so it picks up changes without requiring plugin reload
- **Proxy credentials not exposed to plugins** — `botConfig` constructed in `createPluginApi()` omits `proxy` and `dcc` sections entirely, so SOCKS5 username/password never reach plugins
- **getModLog dynamic SQL is safe** — All WHERE conditions are hardcoded strings (`'action = ?'`, `'channel = ?'`, `'target = ?'`); user values flow only through `?` placeholders
