# Security Audit: Full Codebase

**Date:** 2026-03-26
**Scope:** All `.ts` files in `src/` and `plugins/`

## Summary

The codebase has a strong security posture overall. Input sanitization is applied consistently at the IRC bridge trust boundary, the plugin API is properly frozen and namespaced, database operations use parameterized queries, and credential exposure is controlled. Two warnings and two informational findings were identified.

**Findings:** 0 critical, 2 warning, 2 info

## Findings

### [WARNING] Mode string not sanitized before `raw()` in IRCCommands

**File:** `src/core/irc-commands.ts:151-167`
**Category:** Protocol injection (defense-in-depth)

**Description:** The `sendMode()` fallback path and `sendModeRaw()` both interpolate the `modeString` parameter directly into a `raw()` call without passing it through `sanitize()`:

```typescript
// sendMode fallback (line 155):
this.client.raw(`MODE ${sanitize(channel)} ${mode} ${sanitize(param)}`);
//                                          ^^^^ not sanitized

// sendModeRaw (line 162-163):
const line = `MODE ${safeChannel} ${modeString} ${safeParams.join(' ')}`;
//                                 ^^^^^^^^^^^ not sanitized
this.client.raw(line);
```

While `channel` and `params` are sanitized, `modeString` is not. The public `mode()` method is exposed to plugins via the plugin API (`plugin-loader.ts:446`), so a plugin calling `api.mode('#chan', '+o\r\nPRIVMSG #evil :payload', 'nick')` could inject arbitrary IRC commands.

Currently, all internal callers pass hardcoded mode strings (`+o`, `-b`, etc.) and all plugins use the same pattern. No existing code path passes user input as a mode string. However, the gap violates the security model's defense-in-depth principle.

**Remediation:** Sanitize `modeString` in both methods:

```typescript
// sendMode fallback:
this.client.raw(`MODE ${sanitize(channel)} ${sanitize(mode)} ${sanitize(param)}`);

// sendModeRaw:
const safeModes = sanitize(modeString);
const line =
  safeParams.length > 0
    ? `MODE ${safeChannel} ${safeModes} ${safeParams.join(' ')}`
    : `MODE ${safeChannel} ${safeModes}`;
```

### [WARNING] DCC port connection race (inherent DCC protocol limitation)

**File:** `src/core/dcc.ts:456,482`
**Category:** IRC-specific

**Description:** When a passive DCC CHAT is negotiated, the bot opens a TCP listener on `0.0.0.0` and tells the requesting user which port to connect to. The `server.once('connection')` handler accepts the first connection regardless of source IP:

```typescript
server.listen(port, '0.0.0.0', () => { ... });
// ...
server.once('connection', (socket: Socket) => {
  // Opens a session with the pre-verified user's permissions
  this.openSession(pending, socket);
});
```

An attacker who can observe the CTCP exchange (e.g., on the same IRC channel) and reach the bot's IP could race to connect to the offered port before the legitimate user, obtaining a session with the legitimate user's permissions. The 30-second timeout window and the fact that the port is used only once limits the attack surface.

This is a known limitation of the DCC CHAT protocol — the token mechanism correlates the offer but does not authenticate the TCP connection.

**Remediation:** This cannot be fully fixed without abandoning the DCC protocol. Mitigations:

- Document the risk in `docs/SECURITY.md` so admins can make an informed choice about enabling DCC
- Consider optionally binding to a specific interface rather than `0.0.0.0` (add a `bind_address` config option)
- Consider logging the connecting IP and comparing it against the user's resolved hostname as a best-effort check

### [INFO] Plugin API exposes filesystem paths via `botConfig`

**File:** `src/plugin-loader.ts:516-517`
**Category:** Data leakage (defense-in-depth)

**Description:** The frozen `botConfig` object given to plugins includes `database` (the SQLite file path) and `pluginDir` (the plugins directory path). While neither is a secret, exposing filesystem paths to plugins gives them information that could be used to access the database file directly (bypassing namespace isolation) if they also have filesystem access.

```typescript
botConfig: Object.freeze({
  // ...
  database: this.botConfig.database,   // e.g. "./data/hexbot.db"
  pluginDir: this.botConfig.pluginDir,  // e.g. "./plugins"
  // ...
}),
```

The services password is correctly redacted (line 514 comment), and proxy/DCC configs are omitted entirely.

**Remediation:** Omit `database` and `pluginDir` from the plugin-facing `botConfig`, or replace them with redacted placeholders. Plugins that need to know the plugin directory can derive it from their own file path.

### [INFO] chanmod `getUserFlags` doesn't IRC-lower the channel key

**File:** `plugins/chanmod/index.ts:142-143`
**Category:** Permissions (correctness, fails safely)

**Description:** The `getUserFlags()` helper looks up per-channel flags using the raw channel name from IRC context, but `Permissions.setChannelFlags()` stores keys using `ircLower()`:

```typescript
// chanmod (raw channel name):
const channelFlags = user.channels[channel] ?? '';

// permissions.ts:138 (stored IRC-lowered):
const normalizedChannel = ircLower(channel, this.casemapping);
record.channels[normalizedChannel] = normalized;
```

If the IRC server sends a channel name with different casing than what was used when setting flags (e.g., `#Test` vs `#test`), the channel-specific flags won't be found. This **fails safely** (no flags = no extra permissions granted), so it's a correctness issue rather than a security vulnerability.

**Remediation:** Use `api.ircLower(channel)` for the lookup:

```typescript
const channelFlags = user.channels[api.ircLower(channel)] ?? '';
```

## Passed Checks

- **Input validation:** All IRC input is sanitized at the bridge trust boundary (`irc-bridge.ts`) using `sanitize()` to strip `\r\n`. IRC formatting is stripped before command parsing. Argument counts are validated before array access.

- **Protocol injection:** The codebase uses `irc-framework`'s typed methods (`say`, `notice`, `action`) wherever possible. All interpolated values in `raw()` calls are sanitized (except `modeString` — see finding above). The `sanitize()` utility is correct and used consistently.

- **SQL injection:** All database operations use `better-sqlite3` prepared statements with parameterized queries. The `getModLog` query dynamically builds SQL with `?` placeholders — verified safe. LIKE wildcards in `list()` are properly escaped.

- **Permissions:** Flag checks are enforced by the dispatcher before handler invocation. The `CommandHandler` correctly checks flags for non-REPL sources (including DCC). REPL gets implicit owner access (physical access assumption, documented). The `+n` flag correctly implies all other flags.

- **NickServ verification:** The `chanmod` auto-op correctly awaits NickServ ACC/STATUS verification when `require_acc_for` includes the relevant flag. Verification has a configurable timeout (default 5s) that resolves to `verified: false` on timeout (fails closed). DCC optionally requires NickServ verification.

- **Plugin isolation:** Plugin API objects are `Object.freeze()`-d. Database access is namespaced per plugin at the `BotDatabase` class level. The services password is omitted from the plugin-facing `botConfig`. Plugin names are validated with `SAFE_NAME_RE` to prevent path traversal. Plugin load errors are caught and don't crash the bot.

- **Credentials:** `config/bot.json` is in `.gitignore`. `config/bot.example.json` contains no real credentials (password field is empty string). SASL credentials are not logged. The `identify()` method sends the password only to NickServ, and the logger message says "Sent IDENTIFY to NickServ" without including the password.

- **DoS protection:** Message output uses a token-bucket rate limiter (`MessageQueue`). Queue depth is capped at 500 messages. Long replies are split and capped at 4 lines. CTCP responses are rate-limited to 3 per nick per 10 seconds. Timer binds have a 10-second minimum interval.

- **Hostmask warnings:** `Permissions.warnInsecureHostmask()` correctly detects and logs a `[security]` warning for `nick!*@*` patterns on privileged users.

- **Plugin error containment:** The dispatcher wraps every handler in `try/catch` (`dispatcher.ts:152-159`). Timer handlers are also wrapped. Plugin `teardown()` errors are caught during unload.

- **World-readable config check:** `bot.ts:369-378` checks `stat.mode & 0o004` and exits if the config file is world-readable.

- **Message splitting:** `splitMessage()` correctly splits at word boundaries and caps output at 4 lines with `...` truncation.

- **DCC session limits:** Max sessions, idle timeout, flag checks, and optional NickServ verification are all in place. Pending DCC offers time out after 30 seconds.

- **Path traversal in `.load`:** The `plugin-commands.ts:48` validates plugin names with `/^[a-zA-Z0-9][a-zA-Z0-9_-]*$/` before constructing the path.

## Recommendations

1. **Fix the mode string sanitization gap** (WARNING #1) — straightforward one-line fix in two methods.

2. **Fix the chanmod channel key lookup** (INFO #2) — one-line correctness fix that also improves security posture.

3. **Document the DCC race condition** in `docs/SECURITY.md` so administrators understand the risk when enabling DCC CHAT.

4. **Consider omitting filesystem paths** from the plugin-facing `botConfig` to minimize information available to plugins.
